# kms-demo

Transparent envelope encryption for S3-compatible object storage using HashiCorp Vault Transit as a KMS.

Two proxy implementations вҖ” Java and Go вҖ” sit in front of MinIO and encrypt every object on write, decrypt on read. Both share the same wire format and are fully interoperable.


**Envelope encryption (DEK/KEK):**
- Each object gets a unique 256-bit data encryption key (DEK) generated by Vault.
- The proxy encrypts the object with the DEK using AES-256-GCM.
- Vault wraps the DEK with the key encryption key (KEK) stored in the Transit engine.
- The wrapped DEK is saved in S3 object metadata вҖ” Vault never sees the object's plaintext.

**Wire format** (identical across Java and Go):

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  12-byte nonce  в”Ӯ  ciphertext  +  16-byte GCM auth tag  в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

Object metadata keys:
| Key | Value |
|-----|-------|
| `X-Amz-Meta-Encrypted-Dek` | Vault-wrapped DEK (`vault:v1:вҖҰ`) |
| `X-Amz-Meta-Original-Content-Type` | Original `Content-Type` |

## Quick start

**Prerequisites:** Docker and Docker Compose. No local Java or Go toolchain needed.

```bash
# 1. Clone (submodules included)
git clone --recurse-submodules https://github.com/pchikoian/kms-demo.git
cd kms-demo

# 2. Build both proxy images
make build-all

# 3. Start Vault, MinIO, and both proxies
make up

# 4. Smoke-test both proxies
make test-all
```

Expected output:

```
==> Java proxy (port 8080)
hello from java proxy
==> Go proxy (port 8081)
hello from go proxy
```

### Available make targets

| Target | Description |
|--------|-------------|
| `make build` | Build Java proxy image |
| `make build-go` | Build Go proxy image |
| `make build-all` | Build both images |
| `make up` | Start the full stack in the background |
| `make down` | Stop all containers |
| `make clean` | Stop containers and remove all volumes |
| `make test` | Smoke-test Java proxy (port 8080) |
| `make test-go` | Smoke-test Go proxy (port 8081) |
| `make test-all` | Smoke-test both proxies |
| `make logs` | Follow logs for all services |

## Uploading and downloading objects

Both proxies expose an S3-compatible HTTP API on their respective ports.

### Java proxy (port 8080)

```bash
# Upload a text file
curl -X PUT http://localhost:8080/demo-bucket/hello.txt \
  -H "Content-Type: text/plain" \
  --data "hello from java"

# Download (decrypted transparently)
curl http://localhost:8080/demo-bucket/hello.txt

# Upload a binary file
curl -X PUT http://localhost:8080/demo-bucket/image.png \
  -H "Content-Type: image/png" \
  --data-binary @/path/to/image.png

# Download binary
curl -o downloaded.png http://localhost:8080/demo-bucket/image.png

# Delete an object
curl -X DELETE http://localhost:8080/demo-bucket/hello.txt

# List objects in a bucket
curl http://localhost:8080/demo-bucket/
```

### Go proxy (port 8081)

Same API, different port:

```bash
curl -X PUT http://localhost:8081/demo-bucket/hello.txt \
  -H "Content-Type: text/plain" \
  --data "hello from go"

curl http://localhost:8081/demo-bucket/hello.txt
```

## Using the AWS CLI

Configure a profile pointing at one of the proxies:

```bash
aws configure --profile kms-demo
# AWS Access Key ID:     minioadmin
# AWS Secret Access Key: minioadmin
# Default region:        us-east-1
# Default output format: json
```

Then use it with `--endpoint-url`:

```bash
# Upload
aws s3 cp myfile.txt s3://demo-bucket/myfile.txt \
  --endpoint-url http://localhost:8080 \
  --profile kms-demo

# Download
aws s3 cp s3://demo-bucket/myfile.txt ./myfile-out.txt \
  --endpoint-url http://localhost:8080 \
  --profile kms-demo

# List
aws s3 ls s3://demo-bucket/ \
  --endpoint-url http://localhost:8080 \
  --profile kms-demo

# Delete
aws s3 rm s3://demo-bucket/myfile.txt \
  --endpoint-url http://localhost:8080 \
  --profile kms-demo
```

Switch `--endpoint-url` to `http://localhost:8081` to use the Go proxy instead.

## Verifying encryption

Confirm the proxy stores ciphertext, not plaintext, in MinIO:

```bash
# Write via the proxy
curl -X PUT http://localhost:8080/demo-bucket/secret.txt \
  -H "Content-Type: text/plain" \
  --data "top secret"

# Read back through the proxy вҖ” returns plaintext
curl http://localhost:8080/demo-bucket/secret.txt
# вҶ’ top secret

# Read directly from MinIO (bypasses decryption) вҖ” returns binary ciphertext
curl http://localhost:9000/demo-bucket/secret.txt | xxd | head
# вҶ’ 00000000: a3f2 1c4b ...  (random binary, not "top secret")
```

Inspect the wrapped DEK in the object's metadata:

```bash
curl -I http://localhost:9000/demo-bucket/secret.txt 2>&1 \
  | grep -i "x-amz-meta"
# x-amz-meta-encrypted-dek: vault:v1:AbCdEf...
# x-amz-meta-original-content-type: text/plain
```

## Interoperability

Objects written by either proxy can be read by either proxy. They share the same AES-256-GCM wire format and the same Vault KEK (`demo-kek`).

```bash
# Write via Java proxy
curl -X PUT http://localhost:8080/demo-bucket/cross.txt \
  -H "Content-Type: text/plain" \
  --data "written by java"

# Read via Go proxy
curl http://localhost:8081/demo-bucket/cross.txt
# вҶ’ written by java

# Write via Go proxy
curl -X PUT http://localhost:8081/demo-bucket/cross2.txt \
  -H "Content-Type: text/plain" \
  --data "written by go"

# Read via Java proxy
curl http://localhost:8080/demo-bucket/cross2.txt
# вҶ’ written by go
```

## Vault operations

The Vault UI is available at http://localhost:8200 (token: `root`).

### Inspect the Transit key

```bash
# List Transit keys
curl -s -H "X-Vault-Token: root" \
  http://localhost:8200/v1/transit/keys \
  | python3 -m json.tool

# View the demo-kek key metadata
curl -s -H "X-Vault-Token: root" \
  http://localhost:8200/v1/transit/keys/demo-kek \
  | python3 -m json.tool
```

### Rotate the KEK

Key rotation wraps new DEKs with the new key version. Existing objects continue to decrypt because Vault keeps all previous key versions.

```bash
curl -s -H "X-Vault-Token: root" \
  -X POST http://localhost:8200/v1/transit/keys/demo-kek/rotate

# Confirm version bumped
curl -s -H "X-Vault-Token: root" \
  http://localhost:8200/v1/transit/keys/demo-kek \
  | python3 -m json.tool | grep latest_version
```

### Rewrap existing objects after rotation

After rotation, old objects still decrypt (Vault retains old key versions). To re-wrap their DEKs with the latest key version use `transit/rewrap`:

```bash
ENCRYPTED_DEK=$(curl -sI http://localhost:9000/demo-bucket/secret.txt \
  | grep -i x-amz-meta-encrypted-dek | awk '{print $2}' | tr -d '\r')

curl -s -H "X-Vault-Token: root" \
  -d "{\"ciphertext\":\"$ENCRYPTED_DEK\"}" \
  http://localhost:8200/v1/transit/rewrap/demo-kek \
  | python3 -m json.tool
# вҶ’ new ciphertext with vault:v2:...
```

## Configuration reference

All configuration is via environment variables. Defaults are set for the Docker Compose stack.

| Variable | Default | Description |
|----------|---------|-------------|
| `S3PROXY_LISTEN` | `:8080` | Listen address for the proxy |
| `S3PROXY_VAULT_ADDR` | `http://vault:8200` | Vault server URL |
| `S3PROXY_VAULT_TOKEN` | _(read from `/shared/s3proxy-token`)_ | Vault token |
| `S3PROXY_VAULT_KEY` | `demo-kek` | Transit key name (KEK) |
| `S3PROXY_MINIO_ENDPOINT` | `http://minio:9000` | MinIO endpoint |
| `S3PROXY_MINIO_ACCESS_KEY` | `minioadmin` | MinIO access key |
| `S3PROXY_MINIO_SECRET_KEY` | `minioadmin` | MinIO secret key |
| `S3PROXY_ACCESS_KEY` | _(empty)_ | Client-facing S3 access key (empty = no auth) |
| `S3PROXY_SECRET_KEY` | _(empty)_ | Client-facing S3 secret key |

The Vault token is read from `S3PROXY_VAULT_TOKEN` first. If unset, the proxy reads it from `/shared/s3proxy-token` (a Docker volume populated by the `vault-init` container at startup).

## Development

### Rebuilding after code changes

```bash
# Rebuild and restart only the proxy you changed
make build-go && docker compose up -d --no-deps s3-proxy-go

make build    && docker compose up -d --no-deps s3-proxy
```

### Updating the upstream submodule

```bash
cd s3proxy-upstream
git fetch && git checkout s3proxy-X.Y.Z
cd ..
git add s3proxy-upstream
git commit -m "chore: bump s3proxy-upstream to X.Y.Z"
```

Then update `s3-proxy/pom.xml` to match the new `<version>` and `<jclouds.version>` from the submodule's `pom.xml`.

### Cloning with submodules

```bash
# Fresh clone
git clone --recurse-submodules https://github.com/pchikoian/kms-demo.git

# Already cloned without --recurse-submodules
git submodule update --init --recursive
```

## Security notes

This project is a **demo**. Do not use the defaults in production.

| Topic | Demo | Production |
|-------|------|-----------|
| Vault mode | Dev (in-memory, no persistence) | Integrated storage or HA cluster |
| Vault token | Root token | Scoped policy, short-lived token or AppRole |
| Vault TLS | None | TLS required |
| MinIO credentials | `minioadmin` / `minioadmin` | Strong credentials, IAM policies |
| Object size limit | Unbounded (full object read into memory) | Set `maxSinglePartObjectSize` |
| Audit | None | Enable Vault audit log |

**KEK rotation** вҖ” rotate periodically:
```bash
curl -s -H "X-Vault-Token: root" \
  -X POST http://localhost:8200/v1/transit/keys/demo-kek/rotate
```

**Vault policy** вҖ” a scoped policy is already created by `vault-init` (`s3-proxy-policy`), but the containers use the root token. In production, bind the containers to that policy token instead.
